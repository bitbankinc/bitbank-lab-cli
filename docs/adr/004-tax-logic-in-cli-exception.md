# ADR-004: 税務向け損益計算ロジックを CLI 内に例外として実装する

## ステータス

Accepted

## コンテキスト

2027 年の確定申告（2026 年分・現行総合課税）に向け、税務・会計データ整形機能を追加する
（計画は [`docs/dev/tax-roadmap.md`](../dev/tax-roadmap.md)）。整形の深さは
(1) 生データ出力 (2) 正規化 (3) bitbank 単体の損益計算（取得原価・実現損益）の 3 段階。
段階 2 までは「取得と整形」で現行責務内だが、段階 3 は計算ロジックであり
ADR-002「CLI に分析ロジックを持たない」と衝突する。

一方、税務は「間違えられない」領域である。skills.md の原則
（Skill に分析ロジックのコードを書かない = モデルが計算する）をそのまま適用すると
LLM が損益計算を行うことになり許容できない。計算は決定的なコードで実行し、
CLI Skill（A/B 層向け）と MCP Tool（C 層向け）が**同一ロジック**を使う必要がある。

置き場所の選択肢は 3 つあった:

- (a) CLI 内に例外として実装（`paper` の前例に倣う）
- (b) 第 3 のパッケージ（tax-core）を切り出し、CLI と MCP が双方依存する
- (c) MCP 側にのみ実装する

## 決定

損益計算ロジック（段階 3）を CLI 内に `tax` カテゴリとして実装する。
ADR-002 の例外とし、`paper` サブコマンドの例外前例に倣う。
MCP Tool は CLI をサブプロセス実行して同一ロジックを利用し、
MCP 側に計算を再実装しない。

## 理由

- **二重実装の回避という ADR-002 の本来の目的に整合する。** 計算の実体は 1 箇所に置く
- **`paper` という例外前例と同型。** tax もローカル完結（private GET とローカル計算のみ）で、
  trade エンドポイントには一切触れない
- **検証戦略が最も素直に成立する。** A/B 層が CLI Skill を実データで使い倒して鍛えた
  ロジック「そのもの」を C 層向け MCP Tool が使う。サブプロセス実行なら
  結果の同一性がバイナリレベルで保証される
- **(b) は期限に対して間接コストが大きい。** リリース・バージョニングの管理対象が増え、
  4〜5 ヶ月の期限（12 月頭完成）に見合わない
- **(c) は検証戦略と矛盾する。** CLI Skill で鍛えられない
- **LLM に計算させない。** Skill は CLI コマンドの呼び出し手順のみを記述し、
  数値は tax コマンドが返す確定値をそのまま使う

## 影響

- ADR-002 は本 ADR を根拠とする例外（`tax` カテゴリ）を持つ。
  CLAUDE.md の例外条項と `.claude/rules/commands.md` のカテゴリ表への追記は
  実装着手時（Week 5〜6）に行う
- 課税方式（総合課税 / 分離課税）は tax ロジックの入口パラメータとして最初から持たせる。
  ただし分離課税の実装そのものは制度詳細（特定暗号資産の定義等）が確定するまで着手しない
- tax コマンドの出力には免責文言（確定的な税務判断をしない）を機械可読な形で含める
- 税務の年区切りは JST（1/1〜12/31）であり、「JST は表示・説明用にのみ使う」規約の
  例外となる。`cli/date-utils.ts` に JST 年境界ヘルパーを追加し、規約文言も併せて更新する
