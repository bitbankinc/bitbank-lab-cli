## 1. 製品ポジショニングとスコープ（B案）

### 1.1 位置づけ【方針】

- 本ツールの出力は「**bitbank口座における取引集計・税計算用参考データ**」であり、「税務上の所得金額」ではない。
- 理由【確定】: 総平均法・移動平均法は、納税者が保有する**同一種類の暗号資産全体**（全取引所・全ウォレット横断）について計算する。国税庁の計算例（FAQ 2-8）自体が、A取引所・B取引所の年間取引報告書を**合算**して1つの計算書で計算している。単一取引所のデータのみからは、正確な譲渡原価は原理的に計算できない。
- 外部からの入庫コインの取得価額は入庫時の時価では**ない**（過去の取得価額を引き継ぐ。§8.4）。自己ウォレットへの出庫は譲渡ではなく、納税者全体の保有数量・簿価を減らさない。

### 1.2 参考損益の表示ガード【方針】

```text
guard_reference_pnl(currency, year):
  必須条件（すべて満たす場合のみ「参考損益」を数値表示。満たさない場合は取引集計のみ表示）:
    (a) ユーザーの明示的な確認（アテステーション）:
        「当年および取得価額の繰越に関係する過去年において、bitbank以外の取引所・
         ウォレットで同一銘柄の保有・取引（購入・売却・交換・付与の受領等）がない」
    (b) 当該銘柄に未解決の入庫（DEPOSIT）がない
        - 入庫がある場合、入庫理由と取得価額の由来（§13）が解決済みであること
        - 解決不能な入庫が1件でもあれば、当該銘柄は「参考損益なし（要個別計算）」
    (c) 前年末残高（数量・簿価）が確定していること
        - 当ツールの前年計算結果、またはユーザー入力（入力値はレポートに明示）
  出庫（WITHDRAWAL）がある銘柄:
    - 出庫自体は譲渡ではないため計算は続行できるが、
      「出庫先での売却・使用・決済は本レポートに含まれない」旨の警告を当該銘柄に付す
  ガード不成立時の出力:
    - 取引集計（購入・売却・付与・手数料の数量/金額合計、年間取引報告書相当のデータ）のみ
    - 「参考損益を表示できない理由」を銘柄ごとに明示
```

### 1.3 スコープ外（レポートに明記する事項）

- 他社取引所・ウォレット・DeFi・NFTの損益、bitbank外での決済使用・贈与等
- 所得区分の最終判定（§2）、申告要否の判定（§10）、税額計算

## 2. 所得区分（FAQ 2-2）と本書の適用範囲 ★新設

### 結論

1. 【確定】暗号資産取引により生じた損益は、**原則として雑所得（その他雑所得）**に区分される（FAQ 2-2）。
2. 【確定】ただし、**その年の暗号資産取引に係る収入金額（利益ではなく収入金額）が300万円を超える場合**は、次の区分になる:
   - 暗号資産取引に係る**帳簿書類の保存がある**場合 … 原則として**事業所得**
   - 帳簿書類の保存が**ない**場合 … 原則として**雑所得（業務に係る雑所得）**
   - 帳簿書類の保存があっても、**営利性が認められない場合など**には事業所得該当性を**個別に判断**（FAQ 2-2注）
3. 【確定】暗号資産取引が事業所得等の基因となる行為に**付随**する場合（事業用資産としての保有・決済使用等）は事業所得。
4. 沿革の整理: この300万円・帳簿保存基準は、所得税基本通達35-2の改正（令和4年10月）を受けて**令和4年12月版FAQから存在**する。**令和7年12月改訂では、営利性等による個別判断の注記が追加された**（区分判定の骨格自体に変更はない）。
5. **本書の適用範囲**: 本書§3以降の計算ルールおよび§9（損益通算・繰越）の結論は、**「その他雑所得」に区分されるケースを主対象**とする。事業所得・業務に係る雑所得に該当する場合、必要経費の範囲、帳簿保存・書類添付義務、申告書の記載先、損益通算・（青色申告の場合の）損失繰越の結論が変わるため、本書の結論をそのまま適用してはならない。

### 根拠

- FAQ 2-2「暗号資産取引の所得区分」〔令和7年12月更新〕、所得税基本通達35-1・35-2
- 所得税法27条（事業所得）・35条（雑所得）

### 実装への落とし込み【方針】

```text
所得区分アラート（ツールは判定しない。警告と申告誘導のみ）:
  revenue_proxy = 年間の売却価額合計 + 信用決済差益合計 + 付与・利用料収入合計   # bitbank分のみの下限値
  if revenue_proxy > 3,000,000円:
      warn("暗号資産取引の収入金額が300万円を超えています。帳簿書類の保存の有無等により
            所得区分（事業所得／業務に係る雑所得）が変わる可能性があります。
            本レポートは『その他雑所得』を前提とした参考データです。税理士にご相談ください。")
  ユーザー属性フラグ（任意入力・レポート注記にのみ使用）:
```

## 3. 評価方法（総平均法・移動平均法）と端数処理

### 結論

1. 【確定】個人の法定評価方法は**総平均法**（届出がない場合。法人は移動平均法で個人と逆）。所法48条の2、所令119条の2・119条の5。
2. 【確定】評価方法は**暗号資産の種類（名称）ごと**に選定。「所得税の暗号資産の評価方法の届出書」は、初めて暗号資産（または新しい種類）を取得した年分の**確定申告期限（翌年3/15）まで**に提出（FAQ 2-5）。
3. 【確定】変更は「変更承認申請書」を**変更年の3/15まで**に提出し承認を受ける。年末までに通知がなければみなし承認。3年（相当期間）継続前の変更は却下されうる（FAQ 2-6、所令119の4）。
4. 【確定】年またぎは「年末残高数量」と「年末残高評価額（年末1単位当たり取得価額×数量）」を翌年の年始残高として繰り越す（計算書の「前年の(H)(I)」）。
5. 【確定】計算方法の定義はFAQ 2-4のとおり（v1 §1の定義・公式数値例を維持。ゴールデンテスト: 総平均法 譲渡原価3,106,000円／移動平均法 3,080,200円／計算書例 所得金額2,189,000円）。
6. **端数処理**:
   - 【確定】暗号資産について円未満の端数処理を定めた**法令・通達・FAQの明文はない**。
   - （参考情報）有価証券には1単位当たり金額の1円未満切上げ規定（所令105①）があるが、これは有価証券の規定であり、**暗号資産に当然に適用・準用されるとまではいえない**。切上げ処理は二次資料に見られる一つの実務処理という位置づけにとどめる。
   - 【互換】国税庁の計算書Excel（総平均法用・移動平均法用）の内部の丸め数式は、現行xlsxの直接解析により**確定済み**（→ **付録D `NTA_SHEET_2025_12`**）。総平均法は単価・原価・繰越とも非丸めで最終段のみ収入切捨て/経費切上げ、移動平均法は売却の都度残高価額を円未満切上げ。旧年度シートに関する二次情報（INT切捨て等）は本書から削除した。
   - 【方針】**内部計算はDecimalで丸めずに保持する**（浮動小数点禁止）。丸めは(i)レポート出力層（最終集計値の円未満表示処理）と(ii)Excel互換モードにのみ隔離する。出力層の表示丸めは「所得金額・収入金額計・必要経費計を円未満切捨て表示、内部値は別途保持」とし、Excel解析後に互換モードと整合を確認して確定する。

### 実装への落とし込み

```python
# --- 総平均法（通貨 c、年 Y）: 内部は Decimal・非丸め ---
opening = carryover[c]                  # {qty, cost}
acq  = sum(acquisitions[c, Y])          # {qty, cost}
disp = sum(disposals[c, Y])             # {qty, proceeds}
unit = (opening.cost + acq.cost) / (opening.qty + acq.qty)   # 丸めない
closing_qty  = opening.qty + acq.qty - disp.qty
closing_cost = unit * closing_qty                             # 丸めない
cogs = (opening.cost + acq.cost) - closing_cost               # 差引方式（計算書と同構造）
carryover_next[c] = {closing_qty, closing_cost}
```

```python
# --- 移動平均法（通貨 c、時系列必須）: 内部は Decimal・非丸め（v1の不整合を修正） ---
book = carryover[c]                     # {qty, cost}
for tx in sorted(txs[c], key=(recognition_ts_jst, seq_no)):   # 安定ソート必須
    if tx.kind == ACQUIRE:
        book.cost += tx.cost_jpy
        book.qty  += tx.qty
    elif tx.kind == DISPOSE:
        if book.qty == tx.qty:
            cogs = book.cost                 # 全量処分: 簿価残を全額原価に（端数残留の掃き出し）【方針】
        else:
            cogs = book.cost / book.qty * tx.qty   # その時点の平均単価（丸めない）
        realized += tx.proceeds_jpy - cogs
        book.cost -= cogs
        book.qty  -= tx.qty
carryover_next[c] = {book.qty, book.cost}
# 年末1単位当たり取得価額（報告用）= 12/31に最も近い取得時点の平均単価
```

```text
不変条件テスト（全通貨・全年で機械検証）:
  I1: opening.cost + Σ取得価額 == Σ譲渡原価 + closing.cost        （Decimal で厳密一致）
  I2: qty は常に >= 0。qty == 0 のとき cost == 0
  I3: 交換取引ごとに 支払側 proceeds_jpy == 受取側 cost_jpy        （§6）
  I4: Excel互換モード有効時のみ、丸め起因の I1 乖離を許容し、乖離額をレポートに明示
```

- 【方針】評価方法はグローバル設定＋通貨別上書き（デフォルト: 全通貨 総平均法）。ユーザーの届出状況の確認欄を設ける。
- 【方針】同一タイムスタンプの複数約定は約定ID（`seq_no`）で安定ソート。

### 解釈が分かれる点・不確実性

- 端数処理の公権的な基準は存在しない（上記のとおり非丸め＋出力層処理で対応）。
- 全量処分時の簿価掃き出し（残留端数の処理）は明文規定がなく【方針】扱い。

## 4. 取引手数料・マイナス手数料

### 結論

1. 【確定】**購入時手数料は取得価額に算入**（FAQ 1-5、所令119の6: 「購入の代価＋購入のために要した費用」。設例: 2,000,000円＋550円＝2,000,550円）。
2. 【確定】**売却時手数料は必要経費**（FAQ 2-3: 「売却の際に支払った手数料」）。
3. 【確定】年間取引報告書の「⑩支払手数料」→ 計算書「手数料等」欄で必要経費控除の動線（FAQ 2-8・2-9）。**二重計上防止**: 取得価額に算入済みの購入時手数料を必要経費欄に再掲しない。ツール出力では「取得価額算入済み手数料」と「必要経費手数料」を分離表示。
4. **マイナス手数料（メイカーリベート）**:
   - 【確定に近い前提】該当する法令・通達・FAQは存在しない（2026-07-23時点で確認できず）。
   - 【方針】本製品の標準処理は次の1本とし、**ユーザーによる方式の自由切替は設けない**（年度間で処理を切り替えると期末簿価を操作する余地が生じるため）。個別事情は手動調整機能（監査ログ付き）で対応する。
     - **標準処理**: 受取時に雑所得の収入（「リベート収入」区分）として計上する。日本円受取は受取額、暗号資産受取は受取時の時価で収入計上し、同額を当該暗号資産の取得価額とする（FAQ 1-5⑤・1-7の枠組みとの整合を根拠とする。取得価額・期末簿価の計算に対して中立で、繰越に影響しない）。
   - 本標準処理は**税務監修を経て確定させる**こと（付録B）。

### 実装への落とし込み

```text
fee 正規化:
  buy:  cost_jpy = 約定代金 + max(fee_jpy, 0)
  sell: proceeds_jpy = 約定代金; expense_fees += max(fee_jpy, 0)
  fee < 0:
      rebate_income += abs(fee) の円換算額（暗号資産建てなら受取時時価）
      暗号資産建ての場合は acquisitions に {qty, cost=時価} を追加
  暗号資産建ての正の手数料（例: ETH/BTC 取引のBTC建て手数料）【方針】:
      厳密処理: 当該数量を微小譲渡として処理＋円換算額を取得価額算入/必要経費計上
      簡便処理を採る場合は仕様書に明記（デフォルトは厳密処理を推奨）
```

### 解釈が分かれる点・不確実性

- リベートの所得区分（雑所得か一時所得か）に公権解釈なし。取引量に応じて機械的に発生するリベートは対価性が強く雑所得構成が自然、というのが本製品の整理（【方針】・要監修）。

## 5. 信用取引（レバレッジ取引）

### 結論

1. 【確定】暗号資産信用取引（借入れによる売買）の損益は**決済の日の属する年分**に計上。譲渡原価は**個別法**（建玉ごと対応）。個人に期末みなし決済（法人FAQ 3-1-14）は**ない**（FAQ 2-13、所令119の7）。
2. 【確定】金利・品貸料（FAQ 2-13注、所基通36・37共-22）:
   - 売建て: 受取金利は売付け価額に**含める**、支払品貸料は売付け価額から**控除**
   - 買建て: 支払金利は買付け価額に**含める**、受取品貸料は買付け価額から**控除**
   - → 個人のFAQ上の原則形は「金利・品貸料を決済損益に織り込む」方式。
3. **建玉管理料（レバレッジ手数料）・信用取引手数料**: FAQに明示なし。
   - 【方針】デフォルトは決済損益への織り込み（FAQ 2-13の枠組みと整合、決済年に帰属）。期間費用（支払年の必要経費）方式は年またぎ建玉で帰属年が変わるため、採用する場合は継続適用を条件に設定固定とする。要監修。
4. **ネット実現損益の利用可否**:
   - 【互換】年間取引報告書「⑨損益合計」と国税庁計算書の「信用・証拠金（差益）」（収入金額側）／「（差損）」（必要経費側）欄の存在により、計算書上はネット額での記載が予定されている。
   - 条件付きの結論【方針】: **年間取引報告書の⑨損益合計と定義・対象期間が一致し、未決済建玉の評価損益や現物損益が混入していないことをツールが保証できる場合に限り**、計算書互換出力ではネット額（差益/差損）を使用できる。「事実上公認」といった断定はしない。
   - ツールはネット額出力と併せて、売付け価額計・買付け価額計・金利・品貸料・管理料・手数料の**分解明細**を常に出力する（申告書の総収入金額・必要経費への分解に対応可能とするため）。
5. 【確定】暗号資産の証拠金取引・FX・先物は**申告分離課税（措法41条の14）の対象外**で総合課税（FAQ 2-12）。したがって現物の損益と同一の雑所得内で通算できる（§9）。

### 実装への落とし込み

```text
position = {open_leg, close_leg, interest(受払方向), lending_fee, mgmt_fee, trade_fee}
帰属年 = close_leg（決済約定）の JST 日付の属する年【方針: 決済「日」の解釈を約定日時に固定】
gross:
  sell_price = 売付け約定代金 (+受取金利) (−支払品貸料)
  buy_price  = 買付け約定代金 (+支払金利) (−受取品貸料)
  realized   = sell_price − buy_price − mgmt_fee − trade_fee    # mgmt_fee の織込み位置は設定固定
検算: realized(自前計算) と 取引所提供ネット損益 の突合（許容誤差設定・乖離時警告）
部分決済: open_leg を数量按分。金利・管理料は対応期間・数量で按分（按分規則を仕様書に固定）
年末: 未決済建玉は当年損益に含めない（評価もしない）
```

### 解釈が分かれる点・不確実性

- 建玉管理料の帰属（決済年 vs 支払年）、個人における金利等の発生時費用処理の可否（法人FAQ 3-1-12類似の処理）は明文なし。
- 「決済の日」の約定日/受渡日の別も明文なし（差金決済では実務上ほぼ同一。基準を仕様に明記）。

## 6. 暗号資産同士の交換（ETH/BTC 等）

### 結論

1. 【確定】**課税イベント**。譲渡価額＝**取得した暗号資産の購入価額（取引時の時価の円換算額）**、譲渡原価＝支払側の評価方法による原価（FAQ 1-3）。
2. 【確定・互換】年間取引報告書の仕様（FAQ 2-9③）: 支払側Aの「年中売却金額」＝取得したBの時価、Bの「年中購入数量/金額」＝Bの数量/時価。→ **Aの譲渡収入とBの取得価額に同一の円換算額を使う**（不変条件I3）。
3. 【確定】外貨建て取引は取引時の**TTM**で円換算（FAQ 2-9※）。
4. 円換算レートの具体的な取り方（約定時刻の約定値/仲値/終値、自社レート/外部指数）は**公権的な指定なし**。
   - 【方針】レート決定順: (1) 自取引所のB/JPYペアの約定時点基準レート（直近約定価格）→ (2) 経由換算（B/X × X/JPY、経路をログ）→ (3) 外部フォールバック（明示フラグ）。継続適用を前提に固定し、全件についてレートソース・時点・経路を監査ログに残す。

### 実装への落とし込み

v1 §4の擬似コードを維持（内部非丸め化のみ修正）。不変条件I3（交換取引ごとの支払側proceeds＝受取側cost）を必須テストとする。

### 解釈が分かれる点・不確実性

- 取得側B基準（FAQ設例・報告書仕様と整合）をデフォルトとするが、支払側A時価基準との乖離が大きい流動性の低いペアでは差が出る。採用基準をレポートに明記。
- FAQ 1-3注の「一時的に必要な暗号資産を取得した場合」の除外の射程は不明確。本ツールは全交換を課税イベントとして扱う旨を明記【方針】。

## 7. レンディング利用料・キャンペーン・エアドロップ

### 結論

1. 【確定】マイニング・ステーキング・**レンディング**等により暗号資産を取得した場合、**取得時点の時価**を総収入金額に算入し、要した費用は必要経費（FAQ 1-7）。取得した暗号資産の**取得価額＝取得時点の時価**（FAQ 1-5⑤）。
2. 【確定】収入計上時期は収入の態様に類似する所得の時期に準じる（FAQ 2-1、所基通36-14）。【方針】レンディング利用料・各種付与は**付与（受領）日**基準で計上し、付与日ごとに時価評価する。
3. 【互換】取引所からの無償交付は年間取引報告書上「年中売却金額＝交付時の時価（売却数量なし）」「年中購入数量/金額＝数量/時価」と記載され（FAQ 2-9①）、計算書を通すと時価相当額が収入に、同額が取得価額に計上される。ツールの計算書互換出力はこの流儀に従う。
4. **所得区分**: 受領時時価を収入および取得価額として**記録**するのは上記のとおり確定的だが、**所得区分（雑所得か一時所得か）は年間取引報告書の形式から導けるものではなく、FAQ 2-2および付与の対価性・偶発性等に基づき別途判定される**。
   - 【方針】本製品のデフォルト区分ラベルは「雑所得（暗号資産関連収入）」とし、懸賞・抽選型など対価性の乏しい付与にはイベント種別で「一時所得の可能性」フラグを付けてレポートに注記する（区分の最終判定はしない）。
5. 【確定】分裂（分岐）による取得は取得時に課税なし・取得価額0円（FAQ 1-6）。エアドロップと区別する。

### 解釈が分かれる点・不確実性

- 一時所得該当性の線引きに公権解釈なし。無価値・値付け不能トークンの受領時時価も不明確（`cost=0・収入0・要確認フラグ`で保留リスト化【方針】）。

## 8. 商品決済・低額/無償譲渡・贈与・相続・5%ルール・認識基準 ★新設

### 8.1 暗号資産による商品・サービス購入（決済使用）

- 【確定】暗号資産で商品を購入した場合、保有暗号資産の**譲渡**に当たり、商品価額（税込）が譲渡価額、平均法による譲渡原価との差額が所得（FAQ 1-2）。
- 【確定・互換】年間取引報告書上、(i) 取引所で円転して決済した場合は売却数量・売却金額に現れるが、(ii) **暗号資産そのもので決済した場合は「⑦移出数量」にしか現れない**（FAQ 2-9②）。→ 報告書・取引所データからは**自己ウォレットへの出庫と決済使用を区別できない**。
- 【方針】bitbank単体では取引所内の決済機能がないため、出庫はすべて `WITHDRAWAL` として記録し、「出庫先での売却・使用・決済は本レポートに含まれない」警告を付す（§1.2）。スキーマにはA案拡張用に `PAYMENT` イベント型を予約する:

```text
PAYMENT { t, currency, qty, proceeds_jpy = 商品・サービスの時価(税込) } → DISPOSE として処理
```

### 8.2 低額譲渡・無償譲渡（贈与）

- 【確定】時価の**70%相当額未満**の対価で個人・法人へ譲渡した場合、対価の額に加えて「実質的に贈与したと認められる金額」（時価70%相当額−対価、として差し支えない）を総収入金額に算入（FAQ 2-10）。
- 【確定】贈与（相続人への死因贈与を除く）・遺贈（包括遺贈・相続人への特定遺贈を除く）により暗号資産を移転した場合、**移転時の時価**を総収入金額に算入（FAQ 2-10、所法40）。
- 【方針】板取引は時価で成立するため取引所内では通常発生しない論点だが、スキーマにイベント型を予約: `GIFT_OUT` / `BEQUEST_OUT` / `LOW_PRICE_TRANSFER` / `DONATION`（寄附はFAQ 1-4: 時価が寄附金の額・譲渡損益認識）。これらはA案・手動登録でのみ使用。

### 8.3 取得価額不明時の5%ルール

- 【確定】売却した暗号資産の取得価額が確認できない場合、**売却価額の5%相当額**を取得価額とすることが認められる（FAQ 2-7、所基通48の2-4）。
- 【方針】ツールが自動適用することは**禁止**。次のガードを置く:
  1. まず履歴復元（年間取引報告書の再交付依頼、銀行口座の入出金、取引履歴×公表相場）を案内する（FAQ 2-7の確認手順）
  2. 5%処理は**ユーザーが銘柄・数量・対象売却を明示選択**した場合のみ適用（`costbasis_provenance = DEEMED_5PCT`）
  3. 適用内容（銘柄・数量・売却・適用日）を監査ログに記録し、レポートに明示

### 8.4 贈与・相続・遺贈による取得価額（入庫理由と由来）

- 【確定】取得価額はすべて「受領時の時価」ではない（FAQ 1-5）:
  - ③ 贈与・遺贈により取得（④を除く）… **贈与・遺贈時の時価**
  - ④ **相続人に対する死因贈与、相続、包括遺贈、相続人に対する特定遺贈**により取得 … **被相続人が選択していた評価方法により評価した金額（被相続人側の評価額）を引き継ぐ**
  - 低額譲受けにより取得した場合の取得価額は、対価＋実質贈与認定額の合計（FAQ 2-10注3）
- 【方針】外部からの入庫を一律「時価取得」としないため、入庫理由 enum を必須化する（§13）。`INHERITANCE` は被相続人側評価額の入力を要求する。

### 8.5 収入計上基準（認識基準）とタイムゾーン

- 【確定】暗号資産取引の総収入金額の収入すべき時期は、**原則: 引渡しがあった日**、**選択: 契約（約定）をした日**の属する年分（FAQ 2-1）。
- 【方針】
  - `RECOGNITION_POLICY = DELIVERY_DATE(デフォルト) | CONTRACT_DATE` をユーザー×年度単位で固定（年内の混用禁止）。
  - 取引所内の現物約定は約定≒引渡のため、いずれのポリシーでも約定日時を使用する旨を仕様に明記（実質差なし。ただしポリシー属性は保存する）。
  - 信用取引の帰属は決済約定日（§5）。
  - **年分の判定はJST**で行う。全イベントについて `ts_utc` と `ts_jst` を両方保存し、`year = ts_jst.year`。取引所APIのUnix時刻（UTC基準）をそのまま年判定に使わない。

## 9. 損益通算・損失の扱い（その他雑所得に該当する場合）

### 結論（**本章の結論は所得区分が「その他雑所得」の場合に限る。§2参照**）

1. 【確定】**雑所得内の内部通算は可能**: 暗号資産の売却損は、同一年分の他の暗号資産売却益、信用・証拠金取引の損益（総合課税雑所得）、レンディング等の収入、暗号資産以外の総合課税の雑所得と通算できる。
2. 【確定】**申告分離課税の「先物取引に係る雑所得等」（金商法上のFX等）・株式等の譲渡所得等とは通算不可**（課税方式が異なる別枠）。
3. 【確定】**他の所得（給与等）との損益通算不可**（所法69、FAQ 2-11）。雑所得全体のマイナスは0として扱われる。
4. 【確定】**損失の翌年繰越不可**（現行制度。繰越制度なし・切捨て）。
5. 所得区分が**事業所得**となる場合は損益通算可・青色申告なら純損失の3年繰越可、**業務に係る雑所得**でも書類保存・添付義務等が変わる。本ツールはこれらのケースの計算を保証しない（§2の警告で誘導）。

### 実装への落とし込み【方針】

- 参考損益は負値のまま出力する（`max(0, pnl)` に丸めない。他ソースとの内部通算の余地を消さないため）。
- 固定注記: 「本金額は当社（bitbank）における暗号資産取引の参考損益であり、所得区分が雑所得（その他雑所得）となる場合、他の取引所・ウォレットの損益、暗号資産以外の雑所得と合算した上で雑所得の金額が計算されます。雑所得の損失は給与所得等と通算できず、翌年へ繰り越せません（2026年分・現行制度）。」

## 10. 「20万円ルール」（免責文言用・修正版）

### 結論

1. 【確定】所得税法121条1項の確定申告不要制度。給与収入2,000万円以下・1か所給与・給与全部が源泉徴収（年末調整）対象の給与所得者について、**給与所得・退職所得以外の所得金額の合計額が20万円以下**なら**所得税**の確定申告は原則不要。2か所以上給与の場合は別判定式（タックスアンサーNo.1900の3・注）。
2. 【確定】この「合計額」には、**法令上この判定に含まれない所得は算入しない**: 申告不要を選択した源泉徴収あり特定口座内の株式等譲渡所得、申告不要を選択した一定の配当所得、源泉分離課税の利子所得等。（「給与・退職以外のすべての所得」という表現は不正確なので使用しない）
3. 【確定】判定は「収入」ではなく**所得金額（必要経費控除後）**で、暗号資産以外の副業等の対象所得も合算する。
4. 【確定】**住民税に同種の申告不要制度はない**（市区町村への住民税申告は別途必要）。
5. 【確定】医療費控除等のため**確定申告を行う場合は、20万円以下の所得もすべて申告に含める**（No.1900_qa）。
6. 【確定】給与2,000万円超ほかNo.1900の各類型は金額に関係なく申告必要。

### 免責文言テンプレート（v2）【方針】

> 本レポートの参考損益が20万円以下であっても、確定申告が不要になるとは限りません。所得税の確定申告不要制度（いわゆる「20万円ルール」）は、給与収入2,000万円以下・1か所給与・年末調整済み等の条件を満たす方について、給与所得・退職所得以外の所得金額の合計額（他の取引所・ウォレットでの損益や副業等の所得を含みます。ただし、申告不要を選択した一定の配当所得・源泉徴収あり特定口座内の株式等譲渡所得や、源泉分離課税の所得など、法令上この判定に含まれない所得を除きます）が20万円以下の場合に、所得税の確定申告を不要とする制度です。この場合でも住民税の申告は市区町村へ別途必要です。また、医療費控除等のため確定申告を行う場合には、20万円以下の所得もすべて申告に含める必要があります。詳細は国税庁タックスアンサーNo.1900をご確認ください。

- 根拠URL: https://www.nta.go.jp/taxes/shiraberu/taxanswer/shotoku/1900.htm ／ https://www.nta.go.jp/taxes/shiraberu/taxanswer/shotoku/1900_qa.htm （2026-07-23閲覧）
- 【方針】ツールは申告要否判定を実装しない（給与情報・他の所得を把握できないため）。

## 12. 将来制度（申告分離課税）— 2026年7月時点・**2026年分に影響なし**

### 結論

1. 【確定】令和8年度税制改正法（所得税法等の一部を改正する法律）が**2026年3月31日成立・公布**。分離課税導入の前提となる金商法・資金決済法等改正法（暗号資産を金商法上の金融商品へ移管）が**2026年7月15日に参院本会議で可決・成立**（暗号資産関連規定は公布から1年以内に施行）。
2. 【確定】適用開始は法律上「**金商法等改正法の施行日の属する年の翌年1月1日以後の譲渡**」。金商法改正が2027年中に施行された場合は2028年1月1日以後となるが、**施行日確定後に最終決定される**（2028年開始は現時点の有力な見通しであり確定ではない）。
3. **制度構造**（税制改正大綱・成立法の解説ベース。詳細は政省令・通達待ち）:
   - **特定暗号資産**（金融商品取引業者登録簿に登録されている暗号資産等）を**暗号資産取引業を行う者に対して譲渡等**した場合 … その譲渡等による**譲渡所得等**として、他の所得と分離して**20%**（所得税15%・住民税5%、復興特別所得税別）の申告分離課税。損失の3年間繰越控除・分離枠内の損益通算あり（通算範囲は限定的）。支払調書制度の整備。
   - **特定暗号資産デリバティブ** … 先物取引に係る雑所得等の課税の特例（申告分離）へ追加する方向。
   - **暗号資産投資信託** … 株式等の課税範囲へ追加する方向。
   - **申告分離課税の対象とならない暗号資産の譲渡** … **総合課税の「譲渡所得」として扱う特則**が設けられ、その場合、譲渡所得の50万円特別控除・長期譲渡所得の2分の1課税・他の総合課税所得との損益通算は**適用されない**方向。
4. **断定しない事項**: 海外取引所・DEX・個人間取引等の**具体的な経路・銘柄がどの区分に分類されるか**は、政省令・通達（未公表）により確定する。本書はこれを予断しない。
5. **本ツールへの影響**: 2026年分（および施行スケジュール上、2027年分の見込み）は現行制度（総合課税・原則その他雑所得）のまま。2026年分エンジンに分離課税ロジックは実装しない。
   - 【方針】将来対応フック: 課税年度→課税方式マッピングテーブル、譲渡経路タグ（§13）、取得価額・履歴データの長期保持（移行後も取得価額計算は必要）。
   - レポート注記例: 「暗号資産の課税方式を見直す法改正が成立していますが（2026年7月時点）、本レポートが対象とする2026年分には適用されません。適用開始時期・対象範囲は今後の政省令等で確定します。」

### 根拠（時点明記）

- 令和8年度税制改正大綱（自由民主党・日本維新の会、2025-12-19）本文: 「暗号資産取引業（仮称）を行う者に対して…特定暗号資産…の譲渡等をした場合には、その譲渡等による譲渡所得等については、他の所得と分離して20%…の税率により課税する」

- `DEPOSIT` は `reason` と `costbasis_provenance` が解決するまで `UNRESOLVED_TRANSFER` フラグを保持し、当該銘柄の参考損益をブロックする（§1.2(b)）。

- 集計パイプライン: `Event列 → 税務仕訳（ACQUIRE/DISPOSE/INCOME/EXPENSE） → 銘柄別平均法計算（§3） → レポート`。仕訳変換表は付録A。

|---|---|---|
| 購入行 | `AOn = AO(n-1)+Un`、`ASn = AS(n-1)+Yn`、`AYn = ASn/AOn` | 単価は**非丸め** |
| 売却行 | `AOn = AO(n-1)−AEn`、**`ASn = ROUNDUP(AY(n-1)*AOn, 0)`**、`AYn = AY(n-1)`（据置き） | **売却の都度、残高価額を円未満切上げ** |
| 年末残高 | BI/BM/BR 列が最後の非空 AO/AS/AY を拾い AO41/AS41/AY41 へ転記 | — |
| 売却原価 | `U47 = Y41−AS41` =（繰越＋購入価額計）−年末残高価額（差引方式） | — |
| 所得金額 | `AU49 = SUM(収入)`、`AU50 = SUM(経費)`、`AV47 = AU49−AU50` | **丸めなし**（総平均法用と非対称） |
| 入力検証 | BW〜CA 列: 数量・価額の片方欠落をエラーカウント | — |

**確定事項**:

1. 移動平均法の丸めは「**売却の都度、残高価額（総額）を円未満切上げ**」の 1 箇所のみ。
   単価そのものは丸めない（単価丸め方式ではない）
2. 切上げ後の残高価額が翌年へ繰越されるため、**移動平均法の年またぎ簿価は整数円**
3. 所得金額の最終段に丸めがない（総平均法用の ROUNDDOWN/ROUNDUP と**非対称**）。
   互換モードはこの非対称を再現する
4. 内部 Decimal 非丸め値との乖離は売却 1 回あたり 1 円未満（切上げ分）が累積し得る
   → 不変条件 I4 の乖離報告の対象
5. 取引行の容量は 27 行/シート（×10 シート）。大量取引はこの様式に直接載らないため、
   ツールは「計算書転記用サマリー」粒度の出力で足りる
